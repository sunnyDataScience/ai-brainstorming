# 蘇格拉底 × 第一性原理 — R&D 產品開發整合工作流

> **版本：** v1.0 ｜ **更新：** 2026-04-09 ｜ **定位：** 策略 / 流程 — 產品文件、技術審查、初期發想之共用框架
>
> **相關文件：** `first_principles_playbook.md`（第一性原理與物理量）、`first_principles_sprint_manual.md`（72 小時可執行手冊）、`firstdollar_prd.md`（產品化範例）
>
> **語言原則：** 繁體中文優先；**S** = 蘇格拉底式對齊（語意、定義、邏輯一致性）；**F** = 第一性／工程分解（約束、瓶頸、可執行分解）。兩者為**同一條管線的不同工段**，非兩套互斥方法論。

---

## 1. 為什麼要「綁在一起」

單用 **S**：容易停在「問得很深」但沒有範圍、沒有實驗、沒有交付物。  
單用 **F**：容易在名詞未對齊時就做分解，產出精緻但解錯題的架構或 PRD。

**整合原則：** 先用 **S** 鎖定「我們在解什麼題」，再用 **F** 把題拆到可執行與可驗證，最後用 **S** 做定義漂移與因果的壓力測試，才進入實作與釋出。

---

## 2. 兩種方式的本質（分工，不是人設）

| 維度 | **S（蘇格拉底式）** | **F（第一性／分解）** |
|------|---------------------|------------------------|
| 核心任務 | 澄清意義、前提、邊界；暴露同名異義 | 還原到約束與基元；分解、取捨、路徑與成本 |
| 典型產出 | 可寫進文件的**定義句**、成功條件、非目標 | 分解樹、瓶頸、SLO、遷移／回滾、實驗設計 |
| 失敗模式 | 無限追問、迴避決策 | 把口號當原理；忽略利害關係人與外部性 |
| 與本 repo 物理量對照 | 對齊「誰、痛點、替代方案叫什麼」等**語意** | 對齊 `first_principles_playbook.md` §2 的**人／痛／手工交付／錢**等可驗證單位 |

---

## 3. 共用管線（四段）

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌──────────────┐
│ S1 對齊題目 │ → │ F  分解取捨 │ → │ S2 壓測假設 │ → │ 決策／實驗／交付 │
└─────────────┘    └─────────────┘    └─────────────┘    └──────────────┘
```

| 階段 | 目的 | 收斂規則（不通過則不往下） |
|------|------|----------------------------|
| **S1** | 問題陳述、名詞定義、成功標準、非目標、利害關係人假設 | 每個核心概念有一句**可檢驗**描述（誰／何時／如何觀察到） |
| **F** | 依約束分解方案、找瓶頸、定 MVP 邊界與依賴 | 每條主路徑對應**指標或實驗**；否則標為「待驗假設」 |
| **S2** | 檢查定義漂移、錯把相關當因果、遺漏反例 | 產出**修訂後假設清單 + 決策紀錄**（取捨理由） |
| **執行** | 實作、釋出、觀測 | 週期性自問：「仍在解同一題嗎？」（短 **S**）+ 資源重算（短 **F**） |

## 4. 三類交付場景（文件／審查／發想）

### 4.1 產品文件（PRD、規格、路線圖）

| 區塊 | 主模式 | 寫入文件的句子長什麼樣 |
|------|--------|-------------------------|
| 背景與問題 | **S1** | 「此處的 X 指〔可觀察現象〕，不包含 Y。」 |
| 目標與指標 | **S1 + F** | **S**：對齊「誰的什麼行為」；**F**：能量測、有基線、有時間窗 |
| 使用者與場景 | **S1** | 每 persona 一句「會／不會用」的原因；明列**非目標使用者** |
| 範圍 In/Out | **F** | Out 用約束寫：「因〔瓶頸／依賴〕，本期不做 Z。」 |
| 方案概述 | **F** | 能力樹 + 依賴 + 最長路徑風險 |
| 風險與開放議題 | **S2** | 每項風險對應「若錯，如何被發現」 |

**寫作前 5 分鐘自檢：** 拿掉品牌名，第一段是否仍讀得出「具體問題」？成功定義是否同時含**結果 + 時間 + 可接受成本**？

### 4.2 技術設計審查（Design / Architecture Review）

建議議程（60–90 分鐘）：

1. **S1（10–15m）**：逐條對齊「本設計須滿足的產品／非功能需求」— 定義不一致則先改文件再審。  
2. **F（25–40m）**：介面與資料流、效能／一致性／成本瓶頸、故障與降級、遷移與回滾。  
3. **S2（10–15m）**：是否偷換定義？兩個團隊是否會把同一詞理解成兩套系統？  
4. **結論（5m）**：Approved / Approved with changes / 需補實驗；**每個 open issue 有 owner 與驗收方式**。

精簡 gate：**需求編號 ↔ 設計段落是否一一對應？** 臨界路徑與 SLO 誰負責？失敗時使用者看到什麼？前三名假設各用什麼實驗在兩週內證偽？

### 4.3 開發初期創意發想

**勿在同一時段混「發散聯想」與「拆到原子」。** 建議分場次：

| 場次 | 模式 | 目的與規則 |
|------|------|------------|
| **A 發散** | **S** 輕量 + 聯想 | 量多、角度多；每個點子一句「誰 + 痛 + 情境」；**不**用 F 打斷靈感 |
| **B 聚斂** | **F** 主導 | 3–5 個方向 → 最小驗證、依賴、兩週內可測什麼；產出「若…則…因為…」+ 指標 + 淘汰規則 |
| **C 掃尾** | **S2** | 「我們是否把酷當成有人要？」「成功可否被第三方複核？」 |

與 **72 小時 sprint** 的銜接：發想場次 B 的產出應能對應 `first_principles_sprint_manual.md` 的物理量畫布與「不寫程式先交付」的檢驗；若對齊不了，表示尚未完成 **F** 的收斂或 **S1** 的題目仍模糊。

## 5. R&D 產品開發階段 — 每環節如何導入

以下以常見 **0→1 → 成長期** 皆可對照的階段劃分；你司若用不同 gate 名稱，可直接把下表的「產出物」對應到既有模板。

### 5.1 總覽表

| R&D 階段 | 核心風險 | **S1** 做什麼 | **F** 做什麼 | **S2** 做什麼 | 建議產出物 |
|----------|----------|---------------|--------------|---------------|------------|
| **探索／機會辨識** | 解假問題、名詞漂浮 | 對齊「痛點是否可被觀察」「誰的問題」 | 列出替代方案真名、約束（法規／通路／技術） | 區分「事實／解釋／願望」 | 問題陳述一頁、假設清單 |
| **發想／方案列舉** | 過早收斂或永不收斂 | 發散時只釐清用語，不談可行性排名 | 聚斂時做分解與實驗排序 | 掃描「自嗨敘事」與倫理／合規盲點 | 假設卡片、實驗 backlog |
| **產品定義／PRD** | 範圍膨脹、指標假 | 定義、成功標準、非目標、利害關係人 | In/Out 用依賴與瓶頸寫理由 | 指標是否可被操弄、是否與題目一致 | PRD、指標表、範圍決策紀錄 |
| **技術設計／架構** | 過度設計、與需求脫鉤 | 需求↔設計逐條對齊語意 | 介面、資料流、SLO、成本、故障模式 | 隱含假設與命名歧義 | 設計文、Review 結論、實驗／PoC 清單 |
| **實作／迭代開發** | 題目漂移、債務掩蓋 | 每次大功能前重問「仍在解同一題？」 | 切分里程碑與技術瓶頸、降級策略 | 復盤時檢查因果敘事是否仍成立 | 里程碑、ADR、風險登錄 |
| **測試／驗證／釋出** | 測到規格但沒測到價值 | 對齊「誰在什麼情境下算成功」 | 測試覆蓋關鍵路徑與 SLO、回滾條件 | 失敗案例是否推翻產品敘事 | 測試策略、釋出檢查清單、復盤 |
| **上線後學習** | 數據故事化、路徑依賴 | 指標定義是否隨時間被偷換 | 下一輪資源重算與瓶頸再估 | 組織敘事與現場證據是否一致 | 學習紀要、下一週期 S1 輸入 |

### 5.2 各階段「導入方式」（可當儀式／門檻）

**探索／機會辨識**  
- 開任何研究或訪談前跑 **S1**：受訪者口中的「效率／體驗／智能」各操作化成什麼行為？  
- **F**：替代方案與成本結構列真名（工具、流程、人力），對齊 playbook「workaround 即競品」。  
- **Gate：** 沒有「可聯絡的具體人與痛點表述」→ 不進入 PRD 級別投入。

**發想／方案列舉**  
- 分開「聯想會議」與「取捨會議」；後者才允許 **F** 主導砍項。  
- **S2** 專門問：「若核心假設錯，我們最早何時、用什麼訊號會知道？」

**產品定義／PRD**  
- PRD 模板強制區塊：**定義表**（S1）、**Out of scope 與原因**（F）、**反指標或淘汰規則**（S2）。  
- 與 FirstDollar 類產品對齊時：指標層優先對齊「真訊號」（例如付款）而非代理分數，見 `firstdollar_prd.md` 與 playbook §7。

**技術設計／架構**  
- Review 第一段永遠是 **S1** 需求朗讀；第二段才是 **F** 圖與數字。  
- **S2** 放在結尾：open issue 若只是「再想想」→ 不合格；須改成可驗證句。

**實作／迭代開發**  
- 每個 epic／大功能：**S1** 一句話對齊產品段落；**F** 標注依賴與最長路徑。  
- Sprint 中若發現需求變更，先補 **S1** 再允許改範圍（避免 silent scope creep）。

**測試／驗證／釋出**  
- **S1**：測試劇本是否覆蓋「真實情境敘述」而非只有 API 契約。  
- **F**：負載、故障注入、回滾與監控對齊 SLO。  
- **S2**：上線後若指標好但訪談差（或相反），禁止單一敘事定論，須分開陳述證據。

**上線後學習**  
- 固定節奏：**S2** 問「我們是否仍在解最初那題」；**F** 重算下一週期瓶頸與預算。  
- 輸入回第一階段的「問題陳述一頁」，形成閉環。

## 6. 與現有 first principles 資產的關係

| 本文件管線 | `first_principles_playbook.md` | `first_principles_sprint_manual.md` |
|------------|-------------------------------|-------------------------------------|
| **S1** | 釐清物理量相關名詞（人、痛、替代方案）是否具體 | Hour 0–6 畫布填寫前後的「不可代填」對齊 |
| **F** | §2 四物理量、§3 快狠準公理（做最少有效動作） | 72 小時內行為邊界與關卡 |
| **S2** | §7 反模式（分數、類比當第一性等） | Sprint 結束的硬性退出／前進判準 |

**不變式：** 任何階段若產出無法寫成「可檢驗句」，就不算完成該階段的 **S** 或 **F**；避免會議產出僅止於「共識感」。

---

## 7. 採用建議（最小可行）

1. **選一個現有儀式**（例：週二 PRD 評審、週四 Design Review）嵌入 **S1 開場 10 分鐘**。  
2. **PRD 與設計文模板**各加一欄：「非目標／Out of scope 原因（F）」「核心名詞定義（S1）」。  
3. **發想會**拆成兩場 calendar block，中間至少隔半日，避免認知混戰。
