# 重構清理指令

呼叫 **refactor-cleaner** agent 安全地識別並移除死碼。

## 步驟 1: 偵測死碼

依專案類型執行分析工具：

| 工具 | 偵測內容 | 指令 |
|------|----------|------|
| knip | 未使用的 export、檔案、依賴 | `npx knip` |
| depcheck | 未使用的 npm 依賴 | `npx depcheck` |
| ts-prune | 未使用的 TypeScript export | `npx ts-prune` |
| vulture | 未使用的 Python 程式碼 | `vulture src/` |
| deadcode | 未使用的 Go 程式碼 | `deadcode ./...` |

## 步驟 2: 分類發現

| 層級 | 範例 | 行動 |
|------|------|------|
| **安全** | 未使用的工具函式、測試輔助、內部函式 | 可放心刪除 |
| **小心** | 元件、API 路由、middleware | 驗證無動態 import 或外部使用者 |
| **危險** | 配置檔、進入點、型別定義 | 調查後再處理 |

## 步驟 3: 安全刪除迴圈

對每個安全項目：
1. **執行完整測試套件** -- 建立基準（全綠）
2. **刪除死碼** -- 精確移除
3. **重新執行測試** -- 驗證未破壞任何東西
4. **測試失敗則** -- 立即用 `git checkout -- <file>` 回復並跳過
5. **測試通過則** -- 繼續下一個

## 步驟 4: 處理小心項目

刪除前：
- 搜尋動態 import: `import()`、`require()`
- 搜尋字串引用: 路由名、配置中的元件名
- 檢查是否從公開 API export
- 驗證無外部使用者

## 步驟 5: 合併重複

移除死碼後，尋找：
- 近似重複函式（>80% 相似）-- 合併為一
- 冗餘型別定義 -- 合併
- 無附加值的包裝函式 -- 內聯
- 無用途的 re-export -- 移除間接層

## 步驟 6: 摘要

```
死碼清理
──────────────────────────────
已刪除:   12 個未使用函式
          3 個未使用檔案
          5 個未使用依賴
已跳過:   2 個項目（測試失敗）
節省:     約 450 行已移除
──────────────────────────────
所有測試通過
```

## 規則

- **絕不在未執行測試的情況下刪除**
- **一次一個刪除** -- 原子變更使回復容易
- **不確定就跳過** -- 保留死碼好過破壞生產
- **清理時不重構** -- 分開關注（先清理，再重構）
