## 0. 作業系統：一人公司的硬限制與死法

**把 AI 當成你的產品團隊** — PM、研究員、設計、工程、客服、行銷都外包給工具，自己專注在「洞察、取捨、決策」。

### 0.1 三個硬限制

| 限制 | 問自己 |
| :--- | :--- |
| **時間** | 每天可投入幾小時？ |
| **金流** | 能承受幾個月沒收入？ |
| **分心成本** | 最多能同時跑幾條產品線？（建議 1~2） |

### 0.2 一人公司最常見的三種死法

1. **做太多** — 不是做不出來，是功能堆砌到失焦
2. **沒有付費閉環** — 不是沒流量，是從頭到尾沒人付錢
3. **沒有決策關卡** — 不是工具不夠，是不知道什麼時候該停

> 你要的是「流程」，不是靈感。

## 1. 需求雷達：從噪音中抓可賣的痛

**階段目標：** 找到「有人痛、願意付錢、你做得出來」的題目池。

### 1.1 可觀察的三種市場訊號

| 訊號 | 意義 |
| :--- | :--- |
| 熱門 App 評價差但很紅 | 需求真、產品爛，可切入 |
| 使用者反覆要求某功能 | 付費動機可能存在 |
| 社群/論壇反覆抱怨同件事 | 痛點穩定 |

### 1.2 AI 用法：你當 PM，AI 當研究助理

**輸入：** 一段市場現象描述

**AI 產出：**
- 可能的使用者族群
- 典型情境（Jobs-to-be-Done）
- 替代方案（競品 / 土法煉鋼）

### 1.3 階段產出物

- **Idea Backlog（至少 20 條）**
- 每條附：目標客群、痛點一句話、目前怎麼解、你可以怎麼更好

### 1.4 過關條件

> 至少 3 條痛點反覆出現

## 2. 問題定義：想法變可驗證假設

**階段目標：** 把「我覺得不錯」變成「我能用數據證明」。

### 2.1 問題結構五要素

| 要素 | 問題 |
| :--- | :--- |
| **人群** | 誰？ |
| **情境** | 何時何地？ |
| **痛點** | 卡在哪？ |
| **代價** | 不解決會怎樣？ |
| **現有替代** | 他現在怎麼勉強過？ |

### 2.2 Problem Statement 公式

> 當【某人】在【某情境】時，會因為【某痛點】付出【某代價】，所以他願意用【金額/時間】換取【結果】。

### 2.3 階段產出物

- **Problem Statement**（1 段）
- **Top 3 需求假設**（可被推翻的那種）

### 2.4 過關條件

> 你能說清楚替代成本

## 3. 市場調研：快狠準的訪談與競品拆解

> 成敗分水嶺。把 AI 當市場調研工具，但判斷權在你。

### 3.1 需求訪談（先 10 人就夠）

**核心原則：** 問「他怎麼活下來」，不是「你喜不喜歡我的產品」。

**訪談 7 問（精簡實戰版）：**

1. 最近一次遇到這問題是什麼時候？
2. 當時你怎麼處理？（步驟越細越好）
3. 最煩的是哪一段？為什麼？
4. 如果有人幫你解決，你希望變快多少？
5. 你為此付出過什麼成本？錢 / 時間 / 情緒 / 風險
6. 你試過哪些工具？為什麼不用了？
7. 你願意付多少？（先問替代成本，再問價格）

### 3.2 競品拆解（AI 省 80% 時間）

**讓 AI 生成：**
- 競品矩陣（功能 / 定價 / 渠道 / 評價）
- 用戶差評聚類（常見抱怨 = 需求缺口）
- 切入角度（做更便宜 / 更快 / 更垂直 / 更簡單）

### 3.3 階段產出物

- **Interview Notes + Pain Ranking**
- **Competitor Matrix**
- **你的差異化一句話（Positioning）**

### 3.4 過關條件

> 10 人中 >= 6 人有相同痛點

## 4. PRD Lite：可做可驗證的需求文件

**階段目標：** 控制範圍，讓你跟 AI / 外包 / 自己都對齊。

### 4.1 PRD Lite 結構（1~2 頁）

| 欄位 | 內容 |
| :--- | :--- |
| **目標使用者** | 一句話 |
| **核心場景** | 3 個（條列） |
| **MVP 只做什麼** | <= 3 個功能 |
| **不做清單** | 越明確越好（這是救命的） |
| **成功指標** | 一週內可觀測（例：註冊 → 完成 → 付費） |
| **風險假設** | 3 條（最可能讓你白做的） |

### 4.2 階段產出物

- PRD Lite（Markdown）
- MVP 範圍（scope）+ 不做清單

### 4.3 過關條件

> MVP 功能 <= 3

## 5. MVP 開發：壓成最小可收錢產品

**唯一原則：** 先做能交付結果的版本，不要做完美系統。

### 5.1 MVP 三種形態

| 形態 | 速度 | 成本 | 適用場景 |
| :--- | :--- | :--- | :--- |
| **Landing Page + Waitlist** | 最快 | 最低 | 驗證需求是否存在 |
| **No-code / Low-code Demo** | 中等 | 低 | 跑通核心流程 |
| **Micro-SaaS** | 最慢 | 最高 | 直接收錢 |

### 5.2 MVP 功能結構模板

```
輸入 → 使用者丟什麼
處理 → AI / 規則 / 工作流
輸出 → 他拿到什麼結果
付費牆 → 免費能體驗、付費才爽
```

### 5.3 付費牆設計原則

> 付費用戶兩個好處：**更快、更好**。

- 免費版：能感覺到價值但不夠爽
- 付費版：更快 / 更好 / 更省事 / 更安全

### 5.4 階段產出物

- 可用的 MVP（可 demo）
- 支付方案草案（至少 2 tier）

### 5.5 過關條件

> 使用者能完成一次核心任務

## 6. 上線獲客：讓市場真的看到你

### 6.1 主渠道選擇（只選 1 條，不要貪）

| 渠道 | 特性 |
| :--- | :--- |
| **內容 SEO** | 慢但穩 |
| **短影音 / 社群** | 快但波動 |
| **社群貼文 / 論壇** | 最省錢但要手感 |
| **投放廣告** | 最快驗證付費，但要控成本 |

### 6.2 最小可行行銷（Minimum Viable Marketing）

你只需要：

- **3 篇內容**（解決同一個痛點）
- **1 個 demo 影片**（30~60 秒）
- **1 個 landing page**（清楚價值 + CTA）

### 6.3 階段產出物

- Landing Page + 追蹤（GA4 / PostHog）
- Demo 影片 / 圖文教學
- 3 個投放素材或社群貼文版本

### 6.4 過關條件

> 每天有穩定曝光入口

## 7. 市場驗證：用數據決定繼續、轉向或放棄

**鐵律：** 只看喜歡、收藏、稱讚，沒用。要看錢或強信號。

### 7.1 驗證指標三層（由弱到強）

| 層級 | 指標 | 意義 |
| :--- | :--- | :--- |
| **注意力** | 點擊率、停留、加入候補名單 | 有人看 |
| **行為** | 註冊率、完成率、回訪率 | 有人用 |
| **金流** | 付費率、轉換率、退款率、ARPU | 有人付 |

### 7.2 一週驗證門檻（可直接套用）

| 漏斗節點 | 門檻 |
| :--- | :--- |
| Landing → 註冊 | **> 8%** |
| 註冊 → 完成一次核心任務 | **> 25%** |
| 完成 → 付費 / 預購 / 訂閱 | **> 2%**（早期能到就很不錯） |
| 退款率 | **< 10%**（超過要警覺） |

### 7.3 階段產出物

- 漏斗報表（Funnel）
- 付費轉換報表
- 10 句用戶原話（用來改產品與文案）

### 7.4 過關條件

> 有付費或強付費信號

---

## 8. 迭代擴張：把成功變成可複製流程

### 8.1 迭代優先級（嚴格順序）

```
1. 先修漏水（完成率）
2. 再拉轉換（付費牆 / 定價 / 方案）
3. 最後才擴功能（容易變成垃圾山）
```

### 8.2 一人公司要追求的三件事

1. **一條穩定獲客管道**
2. **一個能自動交付的產品**
3. **一個可重複的上線節奏**（每 1~2 週一個小版本）

### 8.3 階段產出物

- 迭代清單（按影響 x 成本排序）
- SOP（把做對的事寫成流程）
- 知識庫（去識別化案例 + 問答）

### 8.4 過關條件

> 1~2 週一次迭代

## 10. AI 創業幻覺警示

講年入千萬很熱血，但實作時抓住更現實的 KPI：

- **你是不是每週都能更接近「有人付錢」？**
- **你是不是每週都能更快地驗證「這條路不對」？**

> 能快速放棄錯的方向，就是一人公司的超能力。

**三條心法**

| # | 心法 | 含義 |
| :--- | :--- | :--- |
| 1 | **先痛後做** | 痛點不清楚，做什麼都像賭博 |
| 2 | **先小後大** | MVP 越小，上線越快，學得越快 |
| 3 | **先錢後美** | 先驗證付費，再談體驗精緻化 |

## 11. 系統化策略 Prompt 模板

> 以下 Prompt 可直接貼到 ChatGPT / Claude 使用，控制在 4000 字元內。

```
你是一位「AI 1 人公司」的資深產品經理 + 成長行銷 + 商業分析師。
你的任務是把我提供的想法/觀察，系統化拆成：
需求定義 → 市場調研 → PRD Lite → MVP 範圍 → 上線驗證 → 迭代策略，
並用可量化指標做決策關卡（Go / Pivot / Kill）。

【我的背景/限制】
- 我是 1 人團隊，資源有限，想用 AI 加速。
- 目標：用最短時間驗證「有人願意付錢」。
- 限制：同時最多跑 1~2 條產品線；MVP 功能 <= 3；一週內要看到強信號。
- 產出格式要清楚、可直接貼到 Notion/PRD。

【輸入（我會填）】
1) 想法一句話：
2) 我觀察到的市場訊號（留言/抱怨/熱門事件/差評等）：
3) 目標客群猜測：
4) 我能提供的優勢（資源/渠道/能力/資料）：
5) 時間與預算（每週可投入__小時；預算__）：
6) 想要的商業模式（訂閱/一次性/用量/服務）偏好：
7) 不想做的事（例如：不投廣告、不做 App、不做企業客）：

【你要先做的事：多路徑思考，但最後要收斂】
A. 提出 3 個可能的切入角度（不同客群/不同場景/不同定價邏輯）
B. 對每個角度做「可行性評分」(0-10)：
   痛度、頻率、付費意願、可觸達、MVP難度、競品壓力
C. 選出 1 個最適合一人公司先打的「首發切角」，並說明捨棄其他的理由

【核心流程（請依序輸出，每段短、狠、可執行）】

(1) Problem Statement（問題定義）
- 一句話公式：當【誰】在【情境】會因【痛點】付出【代價】，
  所以願意用【錢/時間】換取【結果】。
- 列出 Top 3 假設（可被推翻的那種）

(2) Research Plan（市場調研計畫）
- 需求訪談問題：7 題（可直接拿去問）
- 10 位受訪者去哪裡找（具體渠道）
- 競品拆解：5 個可能替代方案類型（不是只列品牌），以及各自弱點

(3) PRD Lite（1~2 頁結構）
- 目標使用者（1句）
- 核心場景 3 個（條列）
- MVP 只做 3 個功能（用「輸入→處理→輸出」描述）
- 不做清單（至少 8 條，越毒越好）
- 風險假設 3 條（最可能讓我白做的）
- 成功指標（必須一週內可量測）

(4) MVP 方案（給我 3 種形態）
- 方案A：Landing + Waitlist（最快）
- 方案B：No-code demo（跑通流程）
- 方案C：Micro-SaaS（可收錢）
  對每個方案：開發天數估計、工具選型、成本、風險、適用情境
  最後推薦 1 個並說明原因。

(5) 定價與付費牆（以「更快/更好」設計）
- 免費能體驗到什麼？（要能感覺到價值但不夠爽）
- 付費後多得到什麼？（更快/更好/更省事/更安全）
- 2~3 個價位梯度（含命名與適用族群）
- 若是訂閱：月/年方案如何避免太早承諾？
- 退款/爭議策略：用最省心的原則

(6) Go-to-Market（只選 1 條主渠道）
- 選 1 條最可能的渠道（社群/SEO/短影音/投放/合作）
- 產出：3 則貼文/素材腳本（不同角度），1 段 60 秒 demo 文案
- 追蹤事件（Event Tracking）：至少 8 個事件
  （曝光→點擊→註冊→完成→付費→退款）

(7) Validation Gate（Go / Pivot / Kill）
- 一週驗證門檻（數字）：
  Landing→註冊、註冊→完成、完成→付費、退款率
- 若未達標：Pivot 方向（改客群/改場景/改付費牆/改渠道）各 1
- 若達標：下一步擴張策略（先修漏水再拉轉換，最後才加功能）

【輸出格式要求】
- 用 Markdown 標題與清單
- 每段最後加一行「本段結論：___」
- 最後給我：一頁式 Roadmap（Day1~Day7 每天做什麼）
- 不要空話，不要說教；不確定就標註「假設」並告訴我怎麼驗證

現在開始，先從 (A)(B)(C) 收斂切角，然後一路輸出到 (7)。
```

## 附錄：五歲小孩也懂版

想像你在賣一個「解決麻煩的小幫手」：

1. 先去問大家：你現在最麻煩的是什麼？
2. 看他們怎麼自己亂解決（用膠帶黏、用 Excel 硬撐）
3. 你做一個很小的工具，先讓他**省一次麻煩**
4. 如果他覺得真的爽，就會說：「這個我願意付錢」

> 你不用一次蓋大樓。你只要先做一個**能讓人少痛一下**的小東西。
